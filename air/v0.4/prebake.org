#+title: Pre-Baked Image Stores
#+state: complete
#+FILETAGS: :node:core:

* Summary

Import the seed image set into a containerd content store at BUILD
time and ship it in the guest image — boot-time import cost drops to
mounting a filesystem. This is the capability image-bundling
distributions structurally cannot offer: their airgap import happens
on the node because their image IS the node payload; kubenyx's node
payload is a Nix closure, and a containerd store is just more closure.

* Design

** =kubenyx.node.prebakeImages = true= (default false)
At build time, a derivation runs containerd with =--root $out= inside
the sandbox (no KVM, no root: containerd's content/metadata plugins
run fine unprivileged with overlayfs snapshotter replaced by =native=
for the bake; the runtime plugins are disabled), imports
=[pauseImage] ++ seedImages= via ctr, and stops. The result is a
complete =/var/lib/containerd= tree: content blobs + boltdb metadata.

** Guest wiring
The baked store ships in the closure; the guest mounts it as the
overlayfs LOWER layer under =/var/lib/containerd= with a tmpfs upper
(containerd needs write; the bake is immutable). The
=kubenyx-seed-images= unit then has nothing to import for baked
entries — it skips entirely when every entry is baked (the unit
disappears from the boot path).
Snapshotter note: the bake uses the =native= snapshotter; the guest
runtime must read those snapshots — either the guest also runs
=native= (acceptable for volatile test guests; snapshot COW cost is
tmpfs-backed) or the bake is re-checked against overlayfs
compatibility during implementation. Decide empirically; record the
call.

** Acceptance
- A bench leg with a deliberately LARGE synthetic image set (~300MB+
  of distinct layers): boot-time import wall vs prebaked mount, both
  measured in-guest; the prebake side must eliminate ≥90% of the
  import cost.
- Existing seeding tests stay green with prebake OFF (default) and a
  variant with prebake ON runs the same pod successfully.
- Default path byte-identity (prebake off ⇒ nothing changes).

* History
- 2026-07-09: Graduated from byod.org §4 (last staged item); ready.
- 2026-07-09: Implemented (=pkgs/prebake-store.nix=, =modules/node.nix=,
  =guests/microvm.nix=). The bake: containerd 2.3.1 with =--root $out=
  in the sandbox, CRI/runtime/task/NRI plugins disabled, grpc+ttrpc
  sockets owned via config uid/gid (chown-to-root is EINVAL in the
  build userns). =kubenyx.node.prebakeImages= (default false) mounts
  the baked tree as overlay lower under =/var/lib/containerd= with a
  tmpfs upper (=/run/kubenyx/containerd-{upper,work}=) via a oneshot
  gating containerd; the seed unit is ABSENT with prebake on (mkIf
  removes it from attrsOf), and vanishing is total since the bake
  covers the whole set.
- 2026-07-09: Snapshotter call, decided empirically as required.
  (1) Baking SNAPSHOTS is impossible in the sandbox: unpack via the
  walking differ needs a bind mount even for native — mount(2) is
  EPERM for the unprivileged build user ("failed to mount ... fstype:
  bind ... operation not permitted", probed with an import minus
  =--no-unpack=). Store normalization would corrupt them anyway
  (every baked file forced 0444/0555, uid 0 — a native snapshot is
  plain dirs, so 1777 tmp dirs and non-root ownership inside images
  would be destroyed). The bake therefore imports =--no-unpack=:
  content blobs + bolt metadata only, both immune to normalization.
  (2) The guest RUNTIME runs =native= when prebake is on: the kernel
  rejects an overlay upperdir that itself lives on an overlayfs, which
  is exactly what the overlayfs snapshotter would create under the
  mounted store — probed in-guest in =tests/prebake.nix= (the mount
  fails), not assumed. Layers unpack lazily at first use (CRI unpacks
  non-unpacked images on image-store sync — proven by pods running
  from baked images of both seed formats) and every snapshot copy
  lands in the tmpfs upper. Default path keeps overlayfs untouched.
- 2026-07-09: Acceptance closed, all three criteria:
  - Bench (=checks.prebake-bench=, in-guest walls from each guest's
    systemd monotonic clock): 10 distinct 30 MB AES-CTR layers
    (~300 MB, incompressible); boot-time import wall 23.304 s vs
    prebaked overlay mount 0.063 s — 99.7% of the import cost
    eliminated (contract ≥90%). Cluster-ready unregressed by the
    overlay: interleaved firecracker A/B (6 pairs, alternating order,
    in-guest KUBENYX-CLUSTER-READY uptime) paired median −0.17 s

  - Correctness: =local-storage= green with prebake OFF (84.55 s,
