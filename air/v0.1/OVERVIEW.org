* Purpose

This directory holds every Kubenyx planning document, organized by *area* (subject) rather than by phase. The project as a whole is at v0.1; the historical directories once named after phases 2–9 were never separate versions — they were phases of v0.1, and were folded into this tree on 2026-07-15. The [[*Historical phase → area mapping][mapping table]] below locates any doc cited under its old path, and the [[*Phase history][phase history]] section preserves each phase's purpose and closing record.

This OVERVIEW is a navigation aid only — it carries no acceptance criteria and is excluded from =airctl status= counts.

* Shared Motivation

The nixpkgs =services.kubernetes= module is effectively unmaintained and architecturally broken (networked cfssl CA with token auth, single flat CA that makes the aggregation layer impossible, single-master by construction, imperative join flow, kubectl-loop addon manager — see [[file:../context/research/prior-art.md][research: prior art]] for the full evidence). k3s/k0s are fast but are patched/bundled distributions, not stock Kubernetes. Kubenyx fills the empty niche: a declarative, stock-binary, performance-first distribution where =nixos-rebuild switch= is the only lifecycle command.

Performance thesis: stock Kubernetes is slow to start mostly because of how it is *deployed* (image pulls, static pods, raft/fsync, leader-election waits, un-tuned defaults), not because of the binaries themselves. Nix store binaries + systemd + tuned flags + a sqlite-backed datastore option close most of the gap to k3s. See [[file:../context/research/fast-start.md][research: fast start]].

* Phase history

Purpose and closing records of the retired phase directories, preserved from their OVERVIEW.org files.

** Phase 1 — Kubenyx: Drop-in Stock Kubernetes for NixOS

A bootable, single-node-first, multi-node-capable Kubernetes cluster built from *stock upstream binaries* (nixpkgs =kubernetes= 1.36.x), wired as plain systemd services by NixOS modules, delivered as a Nix flake. No kubeadm, no k3s/k0s, no =services.kubernetes= from nixpkgs.

Implementation order was: =flake-packaging= (skeleton) → =pki= → =datastore= → =control-plane=, then =node-runtime= → =networking= → =dns-addons=, with =vm-tests= throughout; a doc is only =complete= when its VM test passes.

- 2026-07-05: Milestone created after the initial research pass.

** Phase 2 — Instant Test Clusters

Phase 2 pushed the testing-first mission (architecture.org, user steer 2026-07-05) past what full VMs can deliver: disposable Kubenyx clusters that appear in single-digit seconds via microVMs, with everything volatile by design.

- 2026-07-06: Milestone opened with the microVM work.
- 2026-07-07: KVM validation executed on EC2 metal: etcd-mem replaces kine (8.5s cluster-ready, single-digit claim now measured, not extrapolated), matrix green, ratio 0.71, snapshot/restore proven at 75ms. kubenyx-snap implementation was the open work item.

** Phase 3 — Durable Clusters

Phase 1 proved stock Kubernetes as NixOS services; phase 2 made disposable clusters instant. Phase 3 extended the same declared-membership model to clusters that are allowed to *matter*: persistent state, an HA control plane, and operator workflows — without taxing the testing-first fast path (every HA branch is gated on =length servers > 1= and the =balanced= profile; single-node testing boots stay byte-identical).

Relationship to phase 2: the multi-node microVM mesh ([[file:microvm/multinode-microvm.org][multinode-microvm.org]]) and this phase share foundations, implemented once during the mesh work — =kubenyx.nodes.<name>.role= (membership knows the server set), the credential bundle format (one tar per node; shipped over the operator channel here, over the tap handoff there), and the endpoint abstraction (a single =controlPlaneEndpoint= string there; a local health-checking LB or external VIP here).

- 2026-07-08: Milestone sketched (draft) alongside the multi-node design discussion, before mesh implementation, so shared schema decisions land once.
- 2026-07-08: durable-ha.org implemented and complete in one campaign: etcd quorum from declared membership, offline-CA custody (6-file bundle incl. the SA keypair), kubenyx-lb with authenticated /readyz probing, hitless agent scale-out proven, all six test legs green, single-node fast path held bit-identical.

** Phase 4 — Bring-Your-Own-Dataplane & Airgap Ergonomics

phases 1–3 built the cluster kubenyx opinionatedly wants: zero-daemon bridge networking, volatile datastores, declared membership. Phase 4 made kubenyx a good HOST for clusters that want different opinions at the dataplane and workload layer — a CNI that enforces NetworkPolicy and encrypts, workloads that claim PersistentVolumes, airgap image sets in standard archive formats — without kubenyx growing any of those opinions itself. Every piece is a hand-off: kubenyx steps aside cleanly, it does not reimplement. The testing-first invariant continued to bind: the defaults are untouched, and the single-node fast path must stay unit-identical.

- 2026-07-08: Milestone opened; byod.org drafted and moved to ready in one step (capabilities and their acceptance tests are fully specified by prior art in the modules).
- 2026-07-08: byod.org implemented and complete same-session: external-CNI mode, storage.localVolumes, archive seeding — all legs green, default path byte-identical. IPv6 + pre-bake remained staged.
- 2026-07-09: ipv6.org implemented and complete: all-v6 single-stack clusters (single + multi-node legs), 29 eval-level CIDR unit tests, v4 default path bit-identical (store-path-level).
- 2026-07-09: prebake.org implemented and complete — the milestone is done. Build-time containerd stores (content + bolt, --no-unpack; sandbox cannot mount, store normalization would corrupt snapshots), overlay-mounted with tmpfs upper, native runtime snapshotter (overlay-upper-on-overlayfs is kernel-rejected, probed in-guest); 99.7% of a ~300 MB set's import cost eliminated, cluster-ready unregressed (paired A/B median −0.17 s), default path bit-identical.

** Phase 5 — Performance Floor

Every prior phase measured performance to prove it did no harm. Phase 5 turned the instruments on kubenyx itself: profile the cold-boot path end to end, attack the ranked costs with paired A/B evidence, and land only what measurably wins. The same discipline applied to the recreation path and the codebase (a cleanliness pass that must keep the whole 15-leg matrix green).

- 2026-07-09: Milestone opened as part of the "complete everything" campaign (prebake + perf + cleanliness + full sweep).
- 2026-07-09: Milestone complete: perf-floor (7.9s -> 3.4s cold boot, ranked attack with paired A/B per keep, bimodality explained as host CPU contention with an enforced bench contract) and cp-growth (learner-based declarative CP scale-out, server-add leg 156s) both landed; 18-leg matrix green.

** Phase 6 — Flexible Testing Strategy

Kubenyx's modules were proven inside kubenyx's own test matrix. Phase 6 turned the repo outward: make embedding a kubenyx cluster in *someone else's* NixOS test harness a first-class, low-boilerplate operation. The phase was driven by a real external embedding exercise — a large consumer test suite hosted kubenyx clusters inside its existing NixOS VM tests — and everything that exercise had to hand-write or work around landed here in generalized form.

- 2026-07-11: Milestone opened; both documents drafted to ready from the embedding exercise's boilerplate/gap inventory.
- 2026-07-11: Milestone complete: lib.harness landed with the =harness= dogfood check (server+agent stood up exclusively through the exported helper), four of five embedding gaps closed byte-identically for defaults (the deliberate agent CoreDNS unit delta is itemized in embedding-gaps.org), 19-leg matrix green. Sweep also surfaced and fixed a latent quiet-host calibration flake in the phase 5 server-add write-loop floor (pre-existing: the pre-phase-6 rev reproduced it). Deferred: etcd-mem stats/range revision semantics.
